Em arquiteturas monolíticas, a responsabilidade pela qualidade dos dados era difusa, levando a problemas como `data drift` e `anomalías de dados` que podem derrubar um modelo em segundos.\n\nÉ aqui que o Datamesh entra com sua proposta radical de descentralização e **produto-ização dos dados**. Mas o Datamesh não exige apenas que os dados sejam produtos; exige que esses produtos venham com um **contrato de qualidade robusto**. E esse contrato, meus amigos, precisa ser **estatístico**.\n\n> [CENA: Slide com a definição de Contratos de Dados Estatísticos. Exemplo: Distribuição Normal $N(\mu, \sigma^2)$]\n\n**VICTOR:** Pensem nisto: um modelo treinado com uma distribuição de dados espera ver essa mesma distribuição em inferência. Se a média, a variância, ou as correlações mudam, o modelo cega. Por isso, um contrato de dados não pode ser apenas sobre tipos de dados (`schema`). Ele precisa especificar propriedades intrínsecas dos dados:\n\n*   **Distribuição:** Não basta dizer que é um número. É um número que segue uma $N(\mu, \sigma^2)$? Ou seus quantis $Q_{25}, Q_{50}, Q_{75}$ estão dentro de certos limites? Se o $\\mu$ ou $\\sigma$ mudam, o modelo pode falhar.\n\n*   **Completude:** Não é só a ausência de `nulls`. É uma taxa de completude superior a 99%, permitindo flutuações dentro de uma banda de tolerância. Matematicamente, $\\text{Completeness}(X) = 1 - \\frac{\\|\\text{nulls}(X)\\|}{\\|\\text{total}(X)\\|}$. Precisamos definir um limiar para isso.\n\n*   **Consistência:** As relações entre colunas. Se duas variáveis, $A$ e $B$, têm uma forte correlação positiva no treinamento, digamos $\\text{Cov}(A, B) > 0.8$, essa relação precisa se manter em produção.\n\n*   **Latência:** A "frescura" dos dados. 95% dos registros devem ter sido gerados nos últimos 5 minutos. Isso é um `Service Level Objective`, um SLO crucial para modelos que dependem de dados recentes.\n\n> [CENA: Slide com "Testes de Hipóteses para Detecção de Data Drift e Anomalias". Equações e explicações breves pro Teste K-S e Qui-quadrado.]\n\n**VICTOR:** Para validar essas garantias em tempo real, empregamos testes de hipóteses que comparam as características dos dados atuais com um *baseline* – geralmente, os dados de treino. Vamos olhar para alguns:\n\n*   **Teste de Kolmogorov-Smirnov (K-S):** Esse é para **distribuições contínuas**. Ele compara a Função de Distribuição Cumulativa (FDC) de duas amostras. A hipótese nula $H_0$ é que elas vêm da mesma distribuição. A estatística de teste K-S é o máximo da diferença absoluta entre as FDCs empíricas:\n    $$ D_{n,m} = \\sup_x |F_n(x) - F_m(x)| $$\n    Um $p$-valor baixo aqui, nos diz que `OPA!`, temos um `data drift` significativo. Os dados não parecem vir da mesma distribuição original. O nosso modelo está em risco!\n\n*   **Teste Qui-quadrado ($\chi^2$):** Para **variáveis categóricas**. Compara as frequências observadas com as esperadas se não houvesse diferença entre as distribuições. A estatística $\\chi^2$ é calculada por:\n    $$ \\chi^2 = \\sum_{i=1}^{k} \\frac{(O_i - E_i)^2}{E_i} $$\n    Novamente, um valor alto de $\\chi^2$ e um $p$-valor baixo indicam um problema, uma mudança na distribuição categórica.\n\n*   **M-estimators (Estimadores de Máxima Influência):** Uma classe de estimadores robustos, menos sensíveis a `outliers`. Pense neles como sentinelas que não disparam um alarme falso a cada ponto fora da curva, mas focam nas mudanças estruturais. São vitais em ambientes ruidosos.\n\n*   **RANSAC (RANdom SAmple Consensus):** Algoritmo iterativo para estimar parâmetros de modelos em dados com `outliers` extremos. Pode ser adaptado para detectar anomalias em séries temporais de métricas de qualidade. Ele busca o *modelo* que melhor se encaixa nos *inliers*, ignorando os *outliers*.\n\n**VICTOR:** A intuição é clara: no Datamesh, a responsabilidade do domínio não termina ao entregar dados, mas sim ao garantir estatisticamente a sua qualidade e previsibilidade contínuas. Sem isso, os "produtos de dados" são apenas um repositório, não uma fonte confiável para ML crítico.\n\n### 2.2 Entropia da Informação e Granularidade em Datalakes\n\n> [CENA: Slide com o título "Entropia da Informação no Datalake" e a pergunta "Por que Guardar Tudo em seu Formato Bruto?"]\n\n**VICTOR:** Agora, vamos virar a moeda e olhar para o Datalake. Isso significa que a informação não é filtrada ou agregada prematuramente, mantendo todas as nuances que futuras análises — especialmente por modelos de ML — possam vir a explorar.\n\n> [CENA: Slide com a Equação da Entropia de Shannon (Contexto Heurístico)]\n\n**VICTOR:** Heuristicamente, podemos pensar na **Entropia de Shannon**, $H(X)$, que quantifica a incerteza ou a quantidade de informação em uma variável aleatória $X$. O **porquê** aqui é combater a infame **"maldição da dimensionalidade"**.\n\n> [CENA: Animação simples mostrando a esparsidade dos dados em dimensões crescentes.]\n\n**VICTOR:** A `maldição da dimensionalidade` significa que, em altos espaços, a distância entre a maioria dos pontos aumenta, tornando os dados esparsos. Algoritmos baseados em vizinhança ou densidade perdem eficiência, e o custo computacional explode.\n\nA álgebra linear nos permite transformar esses espaços complexos em representações mais concisas, significativas e úteis para Machine Learning. É a ponte entre a vastidão do Datalake e a operacionalização dos modelos.\n\n> [CENA: Slide com SVD: $M = U \\Sigma V^T$ e explicação dos componentes.]\n\n**VICTOR:** Comecemos pela **Decomposição de Valor Singular (SVD)**. Dada uma matriz de dados $M$, a SVD a fatora em:\n\n$$ M = U \\Sigma V^T $$\n\nOnde $U$ e $V^T$ são matrizes ortogonais, e $\\Sigma$ é uma matriz diagonal com os `valores singulares` $\\sigma_i$. O **porquê** da SVD é poderoso: ao reter apenas os $k$ maiores valores singulares, obtemos uma **aproximação de baixa patente** de $M$. Isso não só reduz dimensionalidade e ruído, mas também revela **estruturas latentes** nos dados, aquelas características ocultas que não são óbvias.\n\n> [CENA: Slide com PCA.]\n\n**VICTOR:** Do mesmo modo, a **Análise por Componentes Principais (PCA)** é uma técnica de `redução de dimensionalidade` que projeta os dados em um novo espaço com eixos ortogonais, chamados `componentes principais`, que capturam a maior `variância` possível. O **porquê** da PCA é simplificar os dados para modelos de ML sem perder informação crítica.\n\n> [CENA: Slide com a definição de Autovalores e Autovetores: $Av = \\lambda v$.]\n\n**VICTOR:** E no cerne da PCA, estão os **autovalores e autovetores**. Para uma transformação linear $A$, um `autovetor` $v$ é um vetor que mantém sua direção, apenas sendo escalado por um `autovalor` $\\lambda$:\n\n$$ Av = \\lambda v $$\n\nNa PCA, os autovetores da matriz de covariância dos dados nos apontam as direções de maior variação, e os autovalores quantificam essa variação. É a forma como identificamos os eixos mais importantes dos nossos dados.\n\n**VICTOR:** As "novas" dimensões descobertas por SVD e PCA são o que chamamos de **fatores latentes**. Eles representam **características subjacentes** que não são diretamente observáveis, mas que explicam as correlações e os padrões complexos nos dados do nosso Datalake. Para nós, cientistas de dados, esses fatores são ouro para `feature engineering` e para obter *insights* sobre a estrutura interna de vastos volumes de dados.\n\n---\n\n## 3. Imagine que temos uma base de dados de treino e, depois, dados em produção. Queremos saber se a distribuição mudou.\n\n#### Código Python: Detecção de Data Drift\n```python\nimport numpy as np\nfrom scipy import stats\n\n# Dados de referência (e.g., dados de treino ou período estável)\nnp.random.seed(42)\ndata_baseline = np.random.normal(loc=0, scale=1, size=1000)\n\n# Novos dados, simulando dois cenários em produção\n# Cenário 1: Pouco drift, dados com pequenas variações mas estatisticamente semelhantes\ndata_production_no_drift = np.random.normal(loc=0.05, scale=1.02, size=1000)\n# Cenário 2: Drift significativo, uma mudança clara na média\ndata_production_with_drift = np.random.normal(loc=0.5, scale=1.0, size=1000)\n\nprint("--- Teste K-S para Cenário 1 (Sem Drift significativo) ---")\n# stats.ks_2samp compara duas amostras. É o coração do monitoramento de distribuição.\nstatistic_no_drift, p_value_no_drift = stats.ks_2samp(data_baseline, data_production_no_drift)\nprint(f"Estatística K-S: {statistic_no_drift:.4f}")\nprint(f"P-valor: {p_value_no_drift:.4f}")\n\n# Nosso critério de decisão: p-valor < 0.05 (alpha)\nif p_value_no_drift < 0.05:\n    print("Rejeitamos H0: Há evidência de data drift significativo. ALERTA!")\nelse:\n    print("Não rejeitamos H0: Não há evidência de data drift significativo. Tudo ok.")\n\nprint("\\n--- Teste K-S para Cenário 2 (Com Drift significativo) ---")\nstatistic_with_drift, p_value_with_drift = stats.ks_2samp(data_baseline, data_production_with_drift)\nprint(f"Estatística K-S: {statistic_with_drift:.4f}")\nprint(f"P-valor: {p_value_with_drift:.4f}")\n\nif p_value_with_drift < 0.05:\n    print("Rejeitamos H0: Há evidência de data drift significativo. ALERTA! Média alterada.")\nelse:\n    print("Não rejeitamos H0: Não há evidência de data drift significativo. Tudo ok.")\n\n```\n\n> [CENA: Rodando o código e mostrando os resultados no terminal.]\n\n**VICTOR:** Observem o $p$-valor. No primeiro cenário, onde o drift é mínimo, o $p$-valor é alto, indicando que estatisticamente não conseguimos rejeitar a hipótese nula de que as distribuições são as mesmas. Nosso modelo estaria seguro.\n\nMas no segundo cenário, com uma mudança substancial na média, o $p$-valor despenca para virtualmente zero. Isso é um sinal claro: as duas amostras **não vêm da mesma distribuição**. O contrato estatístico de dados foi quebrado, e nosso modelo de ML está operando em um terreno desconhecido. **É por isso que monitoramos!**\n\n### 3.2 Redução de Dimensionalidade com PCA\n\n> [CENA: Tela mostrando o código Python para PCA.]\n\n**VICTOR:** Agora, vamos para o Datalake e o desafio da alta dimensionalidade. A PCA ao resgate!\n\n#### Código Python: Redução de Dimensionalidade com PCA\n```python\nfrom sklearn.decomposition import PCA\nfrom sklearn.preprocessing import StandardScaler\nimport numpy as np\nimport pandas as pd # Usado para contexto, mas não estritamente para PCA aqui\n\n# Simular dados de Datalake com alta dimensionalidade\n# Ex: dados de sensores, features de transações. Criamos correlações propositais.\nnp.random.seed(0)\nX = np.random.rand(100, 10) # 100 amostras, 10 features iniciais\nX[:, 0] = X[:, 1] * 2 + np.random.rand(100) * 0.1 # Feature 0 correlacionada com Feature 1\nX[:, 3] = X[:, 2] * 0.5 + X[:, 5] * 0.8 + np.random.rand(100) * 0.2 # Outras correlações\n\n# MUITO IMPORTANTE: Padronização dos dados antes de PCA. \n# Por que? PCA é sensível à escala. Features com maior variância dominariam os componentes.\nscaler = StandardScaler()\nX_scaled = scaler.fit_transform(X)\n\n# Aplicar PCA para entender a variância explicada\npca = PCA(n_components=None) # n_components=None para ver todos os componentes\npca.fit(X_scaled)\n\nprint("\\n--- Variância Explicada por Componente Principal (PCA) ---")\nprint("Variância explicada por cada componente:", np.round(pca.explained_variance_ratio_, 4))\nprint("Variância explicada acumulada:", np.round(np.cumsum(pca.explained_variance_ratio_), 4))\n\n# Transformar os dados para um espaço de menor dimensão (e.g., 2 componentes)\n# O \'porquê\' de 2 componentes é para simplificar para vizualização ou modelos.\npca_2d = PCA(n_components=2)\nX_reduced = pca_2d.fit_transform(X_scaled)\n\nprint(f"\\nDimensões originais: {X.shape}")\nprint(f"Dimensões após PCA (2 componentes): {X_reduced.shape}")\nprint("Primeiros 5 pontos nos dados reduzidos (2D):\\n", np.round(X_reduced[:5], 4))\n```\n\n> [CENA: Rodando o código e mostrando os resultados no terminal.]\n\n**VICTOR:** Reparem na `variância explicada acumulada`.
